1. Overview

ezMarker is a web application that allows instructors to upload grades into MyLearningSpace with ease. MyLearningSpace (MLS) is Laurier's online learning management system. It is a rebranded version of a product called Brightspace, created by Desire2Learn (D2L). This educational platform allows students and instructors to communicate, share resources, and keep track of student progress. A vital part of this system is a feature that allows instructors to set grades and give feedback on student work. The current feedback system on MLS is tedious and becomes very time consuming when inputting individual feedback for large classes. ezMarker will cut down on this time by automating grade and feedback upload with the use of the Brightspace API.

1.1 Purpose, Scope, and Objectives

1.1.1 Purpose

This document outlines the requirements, goals, and features for the ezMarker application. Throughout the lifecycle of this project, anyone involved in its creation will refer to this document for the basis of their work.

1.1.2 Scope

The software being produced is a web application called ezMarker. This software is used by professors and markers at Wilfrid Laurier University who need to upload student grades and feedback to the university's online learning environment, MLS. Markers that input grades directly through the MLS website can take hours to complete a class. The objective of this project is to provide an alternate user interface that cuts grade recording time down to minutes with batch file uploads.

3. Definitions

Bohr: A server used and maintained by Wilfrid Laurier's Department of Physics and Computer Science to host course information.

Brightspace: A learning management system produced by Desire2Learn. The Brightspace API is used for authentication for entry into the ezMarker website and used to upload student grades in batches.

Desire to Learn (D2L): A Waterloo-based company that produces learning management software. They are the creators of Brightspace.

MyLearningSpace (MLS): Laurier's online learning management system for students and professors. This software is a rebranded version of Brightspace.

4. Project Organization

4.1 External Structure

The client for this project is David Brown. The team leader communicates with this client via email or meetings and relays information to the rest of the team. Another external entity is the MLS website and Brightspace API. The team leader communicates with MLS affiliates via email.

4.2 Internal Structure

The internal structure of the team is based around four distinct divisions. Each division has a leader that attends regular meetings with other group managers to discuss the progress in their division. The project leader represents the entire team and communicates with all external entities.

Each division is expected to organize regular meetings that all members of the division will attend.

Communication with all members of the team and/or division will be done using Slack as the primary tool.

Decisions will be made democratically. Leaders' votes are worth the same as regular members.

4.3 Roles and Responsibilities

Role Responsibilities
Project Leader
  • Attend regular meetings with other team leaders as well as meetings with assigned division
  • Lead communicator with client (David Brown)
  • Lead communicator with MLS affiliates
  • Help other team members to best of their knowledge
Team Leader
  • Attend regular meetings with other team leaders as well as meetings with assigned division
  • Lead their division into executing tasks to the best of their ability
  • Organize teams to deliver work on time as expected
Frontend Members
  • Deliver working code as expressed in project guidelines
  • Create an appealing, user-friendly experience
  • Work with backend team to integrate all parts so they work
Backend Members
  • Deliver working code as expressed in project guidelines
  • Create an easy-to-navigate, working backend
  • Work with frontend team to integrate all parts so they work
Documentation Members
  • Create all required documentation
  • Organize all documentation to be easily navigable
Testing Members
  • Test all code to be correctly working and functioning as intended
  • Notify other team members if work does not meet team standards
  • Document what has been tested and confirm that it is working

In addition to this, all members are expected to:

  • Work with their team to produce the best possible result
  • Meet expected deadlines to deliver work
  • Attend group meetings

5.1.3 Resource Acquisition Plan

The project team has been given all required resources at the beginning of the project. All team members are expected to have:

  • A GitHub account that is in the project repository
  • Git installed on their local machine for version control management
  • A Slack account to communicate with project members
  • An ezEmployee account to record time spent on the project
  • An Asana account to submit tickets and track bug and feature changes
  • Access to appropriate development tools

5.1.4 Project Staff Training Plan

Project staff will be trained through regular attendance of class. A Git training session has been held to teach members how to use version control for this project.

5.2 Control Plan

5.2.1 Requirements Control Plan

All requirement changes require a team discussion to evaluate the change in relation to project guidelines, scope, and quality.

Two main techniques include:

  • Traceability - Any change in the project can be directly traced back to the requirements plan
  • Impact Analysis - Team will discuss impact of change

5.2.2 Schedule Control Plan

Each member of the team will make their own individual efforts to consult the project schedule and meet deadlines. Upon missing a deadline, the schedule will be updated accordingly through discussion among division members. Each team member is expected to record time spent working on each part of the project and evaluate whether they will meet the deadline that is set on the schedule.

5.2.3 Quality Control Plan

Each member of the team is expected to produce work that is correct and maintains the team’s standards of quality. To ensure this, the Testing team regularly reviews code and documentation. They ensure the software is working as intended and take action if it does not meet team standards. They are expected to notify the team that produced the fault by submitting a ticket to Asana.

5.2.4 Reporting Plan

There are 3 main reporting mechanisms used by the project team:

  • Regular meetings - The team regularly gets together and reporting can be done verbally among group members
  • Slack - Reports can be done through the team’s main communication tool
  • Asana - Tickets can be made in Asana that notify members that a report has been made and provide the details of said report

5.2.5 Metrics Collection Plan

The website ezEmployee is used to collect project metrics. Project members are expected to accurately log their time spent working each week on the project.

5.3 Risk Management Plan

Risk factors that the project faces include work that does not meet the deadline due to class conflicts, external responsibilities, and unforeseen circumstances. Conflicts are constantly tracked and if a conflict is met, action is taken to still attempt to meet that deadline.

Integration between frontend and backend may not work as expected and will be met with a meeting both teams. The use of a version control system, Git, allows the team to revert back to working code in the event of a fault-introducing commit.

5.4 Project Closeout Plan

Upon completion of the software project in December of 2017, the client will be given:

  • The final copy of the project
  • All documentation for the project
  • A log of all commits and events from September 2017 to December 2017
  • A log of all hours spent by each team member on the project from September 2017 to December 2017
  • A log of all tickets from September 2017 to December 2017

6.2 Methods, Tools and Techniques

The project is split up into two sections: frontend and backend. They are being worked on by separate teams and are stored in a Github repository. The leaders of each section are to work out internally the preferred strategy and framework for their team.

This software will be made as a web application with no consideration for mobile usability.

Frontend

  • JavaScript 1.8
  • HTML5/CSS

Backend

  • Python 3.6
  • PHP 5.4
  • Brightspace API: Both Python and PHP versions are being developed
  • Apache: A web server with support for PHP, Python3.x, and Flask

Miscellaneous tools to be used: Google Docs, Dropbox, MS Word.

6.3 Infrastructure Plan

The software is a companion application to Brightspace. At key checkpoints prototypes will be uploaded to the test server https://ezmarker.herokuapp.com/. ezMarker will be transferred to Bohr upon completion. Additional resources may include but are not limited to printers, drafting materials, and other University resources.

6.4 Product Acceptance Plan

Working prototypes will be presented to the client at key intervals for assessment along with progress documentation. The client will interact with prototypes on a test server, which will simulate the final product. Internal beta tests will also take place after each unit is completed.

7.3 Quality Assurance Plan

A QA team is dedicated to each stage of development. Members may be interchangeably assigned to tasks depending on the needs of unique cases. Multistage testing is to take place, accompanied by unit testing logs.

7.4 Reviews and Audit Plan

The team is to be distributed prototypes that will be unit tested prior to client presentation. The team is to also peer review code and documentation within their sections and resolve any immediate bugs, along with proper documentation. All features will be presented to the client for acceptance testing before finalization.

7.5 Problem Resolution Plan

In the event of issues, the team is to follow the general contingency plan listed below:

  • Create a ticket in Asana
  • Contact the appropriate parties via Slack

If the issue cannot be resolved and has a detrimental impact on scheduling or the final product, the team leader is to contact the client immediately with possible alternatives.
