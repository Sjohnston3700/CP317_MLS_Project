Overview

ezMarker is a web application that allows instructors to upload grades and feedback onto MyLearningSpace with ease. MyLearningSpace (MLS) is Wilfrid Laurier University's online learning management system. It is a rebranded version of Brightspace, created by Desire2Learn (D2L). This educational platform allows students and instructors to communicate, share resources, and keep track of academic progress. A vital part of this system is a feature that allows instructors to set grades and give feedback on student work. The current feedback system on MLS becomes tedious and is time consuming when giving feedback for large classes. ezMarker cuts down time by allowing bulk grade and feedback upload with the use of the Brightspace API.

A driving force behind reworking the feedback feature is the Autograder program, currently used by the computer science department at Wilfrid Laurier University. Although designed to reduce time spend grading programming assignments, uploading feedback from the Autograder still takes considerable amounts of time using the conventional MLS feedback system. ezMarker cuts down on this time by allowing bulk grade and feedback upload.

ezMarker builds on and extends an existing project developed by Harold Hodgins for the upload of grades and feedback on MLS.

Purpose

The purpose of this document is to organize project deliverables and ensure that the project is completed in a timely fashion and meets the client's needs. This SPMP documents all resources, schedules, checkpoints, and major tasks needed for the completion of ezMarker. The project requirements are collected and listed in a seperate document.

Scope

ezMarker is a website that adds extended functionality to the MLS platform by providing an interface for setting student grades and feedback that is superior to the current MLS implementation. It is branched off of an existing projected completed by Harold Hodgins, which is being upgraded with extended features and improvements of previous functions to fit the new MLS platform smoothly. In the existing system, feedback has to be set individually for each student. For courses with many students, inputting feedback for seperately each student can take hours. ezMarker provides an alternate user interface that reduces grade recording time to minutes by allowing users to upload a single file that contains both grades and feedback for any students the user wishes to mark.

ezMarker also offers the capability to manually input grades and feedback for selected students, such that ezMarker can replace the current MLS website for all marking purposes.

Project Organization

External Structure

The client for this project is Mr. David Brown. The team leader is responsible for communicating with the client and relaying information to the rest of the team. The other external entities are the MLS website and Brightspace. A team member has been appointed to communicate with MLS and Brightspace.

Internal Structure

The internal structure of the team is based around five distinct divisions: front-end, back-end, documentation, QA, and design. Each division has a leader that attends regular meetings with other group leaders to discuss the progress in their division and bring up any issues that they have encounted.

Each division is expected to organize regular meetings that all members of the division are to attend.

Communication within the team is done primarily using Slack. Email communication and in person meetings may take place if needed.

Decisions are made democratically. Leaders' votes are worth the same as regular members. Ultimately, it is the client's decision on whether the team is to proceed with a task.

Control Plan

Requirements Control Plan

All requirement changes require a team discussion to evaluate the change in relation to project guidelines, scope, and quality.

Two main techniques include:

  • Traceability - Any change in the project can be directly traced back to the requirements plan.
  • Impact Analysis - The team is discuss impact of changes prior to any action taken.

Schedule Control Plan

Each member is to make their own individual efforts to consult the project schedule and meet deadlines. Following a missed deadline, the schedule will be updated accordingly through discussion among division members. Each team member is expected to record time spent working on each part of the project, evaluate whether they will be able to meet the deadline, and report to their team leader if they cannot.

Quality Control Plan

Each member of the team is expected to produce work that adheres to the requirements document and maintains the team’s standards of quality. To ensure this, the QA team regularly reviews code and documentation. The team is to ensure the software is working as intended and submit a ticket via Asana if it does not meet project standards.

Reporting Plan

There are 3 main reporting mechanisms used by the project team:

  • Scheduled Leader Meetings - The team leaders meet every Tuesday at 10:00AM. Other members who would like to contribute are encouraged to attend.
  • Slack - Communication is primarily held here, and members are encouraged to discuss matters in their appropriate Slack channels prior to submitting a ticket.
  • Asana - Tickets should be submitted in Asana to notify members that a fault has been noted. Details such as division and actions that may have triggered the error must also be provided of said report.

Metrics Collection Plan

The website ezEmployee is used to collect project metrics. Project members are expected to accurately log their time spent working each week on the project.

Risk Management Plan

Risk factors that the project faces include work that does not meet the deadline due to class conflicts, external responsibilities, and unforeseen circumstances. Conflicts are constantly tracked and if a conflict is met, action is to be taken to attempt to meet that deadline.

Integration between front-end and back-end may not occur as anticipated and will be discussed in a meeting between the front-end and back-end developers.

The use of a version control system, Git, allows the team to revert back to working code in the event of a fault-introducing commit.

Methods, Tools and Techniques

The project is split up into two sections: front-end and back-end. They are being worked on by separate teams and are stored in a Github repository. The leaders of each section are to work out internally the preferred strategy and framework for their team.

This software is developed as a web application with no consideration for mobile usability.

Front-end:

  • JavaScript 1.8
  • HTML5/CSS

Back-end:

  • Apache (A web server with support for PHP, Python3.x, and Flask)
  • Brightspace API
  • PHP 5.4
  • Python 3.6
  • Flask

Miscellaneous tools to be used: Google Docs, Dropbox, MS Word.

Infrastructure Plan

The software is a companion application to Brightspace. At key checkpoints, prototypes are uploaded to the test server, and transferred to Bohr upon completion. It is to be supported by the Flask framework for the Python verions. Additional resources may include but are not limited to printers, drafting materials, and other University resources.

Product Acceptance Plan

Working prototypes are presented to the client at key intervals for assessment along with progress documentation. The client interacts with the prototypes on a test server, which simulates the final product. Internal beta tests are also taking place after each unit is completed.

Quality Assurance Plan

A QA team is dedicated to each stage of development. Members may be interchangeably assigned to tasks depending on the needs of unique cases. Multistage testing is to take place, accompanied by unit testing logs.

Reviews and Audit Plan

Prototypes of the product is to be distributed to the team to be thoroughly unit tested prior to client presentation. The team is to also peer review code and documentation within their sections and resolve any immediate bugs, along with proper documentation. All features are presented to the client for acceptance testing before finalization.

Problem Resolution Plan

In the event of critical issues, the team is to follow the general contingency plan listed below:

  • Create a ticket in Asana.
  • Contact the appropriate parties via Slack.

If the issue cannot be resolved and has a detrimental impact on scheduling or the final product, the team leader is to contact the client immediately with possible alternatives.
